Obsidian Plugin Feasibility Assessment

Live Taildown Rendering in Obsidian

Project Scope: Development of an Obsidian plugin that provides live rendering of .td files with the same interactive experience as Obsidian's native Markdown preview.

Executive Summary

Building an Obsidian plugin for Taildown with live rendering is feasible but complex, requiring approximately 120-180 development hours spread across 6-8 weeks. The plugin would enable users to write .td files in Obsidian with real-time rendering of all Taildown features including components, styling, and interactive elements.

Complexity Rating: High (7/10)

Recommended Approach: Phased implementation starting with basic rendering, then adding syntax highlighting, followed by interactive components.

Current Taildown Architecture Analysis

Core Compiler System

The Taildown compiler has a sophisticated multi-stage pipeline:

Pipeline Stages

  1. Parser ‚Üí MDAST with custom directive nodes
  2. Style Resolver ‚Üí Plain English ‚Üí CSS classes
  3. Component Handlers ‚Üí Interactive component structures
  4. HAST Generator ‚Üí HTML-ready tree with classes
  5. CSS Generator ‚Üí Tree-shaken Tailwind-like CSS
  6. JS Generator‚Üí Vanilla JS for interactivity

Key Technical Details:

  • Performance: Sub-linear scaling, ~50 nodes/ms
  • Dependencies: unified, remark-parse, remark-gfm, mdast-util-to-hast
  • Output: HTML (2-10KB), CSS (5-20KB), JS (2-5KB when needed)
  • Components: 18 standard components (12 static, 6 interactive)

Component Inventory

Component Type Count Interactive Examples
Static Layout 6 No card, grid, container, alert, badge, button
Interactive UI 6 Yes tabs, accordion, carousel, modal, tooltip
Navigation 4 Partial navbar, sidebar, breadcrumb, pagination
Display 2 No avatar, progress

Interactive Components Use:

  • Data attributes (data-component, data-tab, etc.)
  • Event delegation pattern
  • ARIA roles and keyboard navigation
  • Vanilla JavaScript (no framework dependencies)

Obsidian Plugin Requirements

1. Syntax Highlighting

Technology: CodeMirror 6 Language Extension

Scope:

  • Define grammar for Taildown syntax (:::, {}, :icon[], inline attributes)
  • Syntax highlighting for component names, attributes, shorthands
  • Distinguish between Markdown and Taildown-specific syntax
  • Proper nesting and scope handling

Complexity: Medium (3/10)

Estimated Time: 15-20 hours

Challenges:

  • CodeMirror 6 has complex state management for highlighting
  • Need to handle nested Taildown components properly
  • Inline attributes need context-aware highlighting

Reference: Similar to existing plugins like obsidian-language-plugin-counter or custom language modes.

2. Reading Mode Rendering

Technology: MarkdownPostProcessor API

Scope:

  • Register post-processor for .td files
  • Parse Taildown source to AST using existing compiler
  • Render HTML from AST
  • Inject generated CSS into document
  • Initialize interactive JavaScript behaviors

Complexity: Low-Medium (4/10)

Estimated Time: 25-35 hours

Implementation Approach:

// Pseudo-code structure
class TaildownPlugin extends Plugin {
  async onload() {
    this.registerMarkdownPostProcessor(async (el, ctx) => {
      // Only process .td files
      if (!ctx.sourcePath.endsWith('.td')) return;
      
      // Compile Taildown
      const source = await this.app.vault.read(ctx.sourcePath);
      const result = await compile(source, {
        inlineStyles: true,
        inlineScripts: true
      });
      
      // Replace element content
      el.innerHTML = result.html;
      
      // Inject CSS
      this.injectStyles(result.css);
      
      // Initialize interactive components
      this.initializeInteractiveComponents(el);
    });
  }
}

Challenges:

  • Need to bundle entire Taildown compiler (~200KB unminified)
  • CSS injection needs proper scoping to avoid conflicts
  • JavaScript behaviors need to work in Obsidian's sandboxed environment
  • Handle file updates/reloads gracefully

3. Live Preview Mode

Technology: EditorExtension + ViewPlugin

Scope:

  • Real-time rendering as user types
  • Maintain cursor position during updates
  • Incremental rendering (only update changed sections)
  • Split view: source on left, preview on right OR inline rendering

Complexity: High (8/10)

Estimated Time: 50-70 hours

Implementation Approaches:

How it works: Source editor on left, live preview on right (like Obsidian's default)

Pros:

  • Cleaner implementation
  • No editor interference
  • Full-featured rendering

Cons:

  • Takes more screen space
  • Not truly "inline"

Complexity: Medium (5/10)

How it works: Render components inline in editor, hide syntax when not editing

Pros:

  • Matches Obsidian's native Markdown experience
  • More intuitive

Cons:

  • Very complex with CodeMirror 6
  • Requires custom decorations and widgets
  • Cursor positioning challenges

Complexity: Very High (9/10)

How it works: Inline preview for simple elements, split pane for complex components

Pros:

  • Best of both worlds
  • Practical balance

Cons:

  • More code to maintain
  • Decision logic needed

Complexity: High (7/10)

Recommended: Start with Approach A (split pane), migrate to Approach C in v2.0.

Challenges:

  • Performance: Live compilation must be <50ms to feel responsive
  • Incremental updates: Full recompile on every keystroke is too slow
  • State management: Track rendered components and update only changed ones
  • Scroll sync: Keep preview scroll position synced with editor
  • Interactive components: Handle component state across re-renders

4. CSS Management

Technology: Custom Style Injection

Scope:

  • Generate Tailwind-like CSS from used classes
  • Scope CSS to .td file views only (avoid leaking to rest of Obsidian)
  • Handle dynamic class additions
  • Support Obsidian themes (light/dark mode)

Complexity: Medium (5/10)

Estimated Time: 15-20 hours

Implementation:

class StyleManager {
  private styleElement: HTMLStyleElement;
  
  injectStyles(css: string, fileId: string) {
    // Scope all CSS to specific file view
    const scopedCSS = this.scopeCSS(css, `[data-td-file="${fileId}"]`);
    
    // Inject or update <style> tag
    if (!this.styleElement) {
      this.styleElement = document.createElement('style');
      document.head.appendChild(this.styleElement);
    }
    this.styleElement.textContent += scopedCSS;
  }
  
  scopeCSS(css: string, scope: string): string {
    // Parse and prefix all selectors
    return css.replace(/([^\{\}]+)\{/g, `${scope} $1 {`);
  }
}

Challenges:

  • CSS class names conflict with Obsidian's styles
  • Need to handle !important rules carefully
  • Theme switching requires CSS regeneration
  • Glassmorphism and animations need vendor prefixes

5. Interactive Component Behaviors

Technology: Vanilla JavaScript Execution

Scope:

  • Execute generated JavaScript for tabs, accordion, carousel, modal, tooltip
  • Manage component lifecycle (mount/unmount)
  • Handle keyboard navigation and accessibility
  • Ensure behaviors don't conflict with Obsidian's shortcuts

Complexity: Medium-High (6/10)

Estimated Time: 20-30 hours

Challenges:

  • Event delegation needs careful cleanup on unmount
  • Modal backdrop must not block Obsidian UI
  • Tooltips positioning relative to Obsidian's viewport
  • Carousel swipe events might conflict with Obsidian's gestures
  • Tab keyboard navigation vs. Obsidian's shortcuts (Ctrl+Tab)

Mitigation Strategies:

  • Use namespaced event listeners (td-tab-switch instead of generic events)
  • Add data-td-component attributes for scoping
  • Capture keyboard events only when focus is within component
  • Provide plugin settings to disable conflicting features

6. File Type Registration

Technology: Custom View Type

Scope:

  • Register .td file extension
  • Custom file icon in file explorer
  • Override default Markdown handler for .td files
  • File creation templates

Complexity: Low (2/10)

Estimated Time: 5-10 hours

Implementation:

class TaildownView extends TextFileView {
  getViewType() { return 'taildown-view'; }
  getDisplayText() { return this.file?.basename || 'Taildown'; }
  
  async onLoadFile(file: TFile) {
    // Load and render .td file
    const content = await this.app.vault.read(file);
    await this.render(content);
  }
  
  async onUnloadFile() {
    // Cleanup
    this.cleanup();
  }
}

Technical Challenges Deep Dive

Challenge 1: Bundling the Compiler

Problem: The Taildown compiler is ~200KB unminified with dependencies

Impact: Plugin size, load time

Solutions:

  • Use esbuild or Rollup to tree-shake unused code
  • Externalize common dependencies if Obsidian provides them
  • Lazy-load interactive component behaviors
  • Target minified bundle ~80-100KB

Complexity: Medium

Effort: 8-10 hours

Challenge 2: Performance at Scale

Problem: Large .td files (>1000 lines) need fast rendering

Current Performance:

  • ~11ms for 577 nodes
  • ~22ms for 1153 nodes

Target: <50ms total for live preview feel

Solutions:

  • Implement incremental rendering (only re-render changed sections)
  • Use virtual scrolling for large documents
  • Debounce input (300ms delay after typing stops)
  • Cache compiled AST and only recompile changed blocks

Complexity: High

Effort: 15-20 hours

Challenge 3: CSS Isolation

Problem: Generated Tailwind classes might conflict with Obsidian's styles

Examples:

  • .button class might already exist in Obsidian
  • .text-primary could conflict with theme colors
  • Animations might be overridden

Solutions:

  • Prefix all Taildown classes with td- (e.g., .td-button, .td-text-primary)
  • Use CSS scoping with [data-td-file] attribute
  • Increase specificity for Taildown styles
  • Test with popular Obsidian themes

Complexity: Medium

Effort: 10-15 hours

Challenge 4: Interactive Component State

Problem: When user edits .td file, how do we preserve component state?

Example: User has tabs component open to "Tab 3", types a character, re-render wipes state back to "Tab 1"

Solutions:

  • Track component state in plugin memory (WeakMap by element)
  • Before re-render, snapshot all component states
  • After re-render, restore states to matching components
  • Provide data-td-state-key attribute for persistence

Complexity: High

Effort: 12-15 hours

Challenge 5: Markdown Parity

Problem: Obsidian has many Markdown extensions (wiki links, embeds, tags)

Impact: Users might want to use Obsidian features inside .td files

Solutions:

  • Phase 1: No Obsidian features, pure Taildown
  • Phase 2: Pre-process to convert Obsidian syntax before Taildown compilation
  • Phase 3: Extend Taildown parser to support wiki links, embeds

Complexity: High (for Phases 2-3)

Effort:

  • Phase 1: 0 hours (explicitly out of scope)
  • Phase 2: 20-30 hours
  • Phase 3: 40-60 hours

Recommendation: Launch without Obsidian-specific features, add based on user feedback

Alternative Approaches

Option A: Full Plugin (Recommended)

Pros:

  • Complete Taildown experience
  • Native feel in Obsidian
  • Full feature parity

Cons:

  • Significant development effort
  • Ongoing maintenance

Effort: 120-180 hours

Option B: External Preview Server

Approach: Plugin opens external browser window with live preview via localhost server

Pros:

  • Much simpler (30-40 hours)
  • No rendering complexity
  • Full CSS/JS support

Cons:

  • Not integrated
  • Requires external window
  • Less convenient

Effort: 30-40 hours

Option C: Export Command Only

Approach: Plugin adds command to export .td ‚Üí .html (no live preview)

Pros:

  • Very simple (10-15 hours)
  • Zero rendering complexity

Cons:

  • No live experience
  • Limited value-add

Effort: 10-15 hours

Conclusion

Creating an Obsidian plugin for Taildown with live rendering is ambitious but achievable. The existing Taildown compiler provides a solid foundation, but significant integration work is required to create a polished, performant plugin.

Bottom Line

:::card{glass padded-lg} Feasibility: YES - Technically feasible with known solutions to challenges

Effort: 120-180 hours - Substantial but reasonable for impact

Risk: Medium - Manageable with phased approach

Recommendation: PROCEEDwith Phase 1 proof of concept

The Obsidian user base is large and engaged, making this a high-value target for Taildown adoption. Starting with basic rendering and iterating based on user feedback is the prudent approach.
